`BroccoliQueue` is the central type in the Broccoli library. It provides a unified async interface for all queue operations — publishing messages, consuming them, acknowledging success, rejecting failures, and inspecting queue state — across all supported broker backends.

## The Builder Pattern

You create a `BroccoliQueue` using the fluent builder API. Call `BroccoliQueue::builder(url)`, chain configuration methods, then call `.build().await` to connect:

```rust
use broccoli_queue::queue::{BroccoliQueue, RetryStrategy};

let queue = BroccoliQueue::builder("redis://localhost:6379")
    .pool_connections(5)
    .failed_message_retry_strategy(
        RetryStrategy::new().with_attempts(3)
    )
    .enable_scheduling(false)
    .build()
    .await?;
```

### Builder Configuration Options

| Method | Parameter Type | Default | Description |
|--------|---------------|---------|-------------|
| `pool_connections(n)` | `u8` | 10 | Number of connections to maintain in the connection pool |
| `failed_message_retry_strategy(s)` | `RetryStrategy` | 3 retries, retry enabled | Retry configuration for failed messages |
| `enable_scheduling(bool)` | `bool` | false | Enable support for delayed/scheduled messages |

For SurrealDB, an alternative constructor accepts an existing connection:

```rust
#[cfg(feature = "surrealdb")]
let queue = BroccoliQueue::builder_with(existing_surreal_db_connection)
    .build()
    .await?;
```

## Message Lifecycle

Messages in Broccoli flow through a well-defined lifecycle with three queue states:

```
Publish
   ↓
Main Queue          ← messages waiting to be consumed
   ↓  consume()
Processing Queue    ← message is in-flight, held atomically
   ↓
├─ acknowledge() → message removed (successfully processed)
└─ reject()      → retry or move to {topic}_failed queue
```

### Queue States Explained

- **Main Queue** (`{topic}`): Published messages wait here until consumed.
- **Processing Queue** (`{topic}_processing`): After `consume()`, the message is moved here atomically. If a consumer crashes without acknowledging, the message remains here for recovery.
- **Acknowledged**: Calling `acknowledge()` permanently removes the message from the processing queue.
- **Rejected**: Calling `reject()` removes the message from the processing queue. If the retry limit has not been reached, it re-queues to the main queue with an incremented `attempts` count. Otherwise it moves to `{topic}_failed`.

## Checking Queue Size

Use `size()` to get the current message count for a queue. This method is available without the `management` feature:

```rust
let sizes = queue.size("jobs").await?;

for (queue_name, count) in &sizes {
    println!("{}: {} messages", queue_name, count);
}
```

`size()` returns a `HashMap<String, u64>`. For standard queues, it contains one entry. For fairness queues (with disambiguators), it contains one entry per disambiguator sub-queue.

## Publishing Messages

Publish a single message to a topic:

```rust
use broccoli_queue::queue::BroccoliQueue;
use serde::{Serialize, Deserialize};

#[derive(Clone, Serialize, Deserialize)]
struct JobPayload {
    id: String,
    task_name: String,
}

let payload = JobPayload {
    id: "job-123".to_string(),
    task_name: "send_email".to_string(),
};

let message = queue.publish("jobs", None, &payload, None).await?;
println!("Published with task_id: {}", message.task_id);
```

## Consuming Messages

Consume a message (blocks until one is available), then acknowledge or reject it:

```rust
let message = queue.consume::<JobPayload>("jobs", None).await?;
println!("Got: {:?}", message.payload);

// Acknowledge on success
queue.acknowledge("jobs", message).await?;

// Or reject on failure (triggers retry)
// queue.reject("jobs", message).await?;
```

## Processing with Handlers

For most use cases, `process_messages()` is simpler than manual consume — it handles the consume/acknowledge/reject cycle automatically:

```rust
use broccoli_queue::brokers::broker::BrokerMessage;

queue.process_messages(
    "jobs",
    Some(4),   // concurrency: 4 parallel workers
    None,      // consume_options
    |message: BrokerMessage<JobPayload>| async move {
        println!("Processing: {:?}", message.payload);
        Ok(())
    },
).await?;
```

On handler success, the message is automatically acknowledged. On handler error, it is automatically rejected.
