In Broccoli, every message you publish or consume is wrapped in a `BrokerMessage<T>` struct that adds important metadata for processing, tracking, and retry management. Understanding the message structure is key to building reliable message-driven systems.

## BrokerMessage Structure

The `BrokerMessage<T>` struct encapsulates your payload along with metadata:

| Field | Type | Description |
|-------|------|-------------|
| `task_id` | `uuid::Uuid` | Unique identifier auto-generated for each message |
| `payload` | `T` | Your message content (must be JSON-serializable) |
| `attempts` | `u8` | Number of processing attempts made so far |
| `disambiguator` | `Option<String>` | Optional fairness queue key for message ordering |

The `metadata` field is `pub(crate)` — it is an internal implementation detail not accessible from user code.

### Example: Accessing Message Fields

```rust
use broccoli_queue::brokers::broker::BrokerMessage;
use serde::{Serialize, Deserialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
struct EmailPayload {
    to: String,
    subject: String,
    body: String,
}

// When you consume a message:
let message: BrokerMessage<EmailPayload> = queue.consume("email_queue", None).await?;

println!("Task ID: {}", message.task_id);
println!("Attempts: {}", message.attempts);
println!("Payload to: {}", message.payload.to);
```

## Payload Requirements

Your payload type `T` must implement the following traits:

- `Clone`: Messages may be cloned during processing
- `Serialize`: For JSON serialization to the broker
- `serde::de::DeserializeOwned`: For deserialization from the broker

The standard way to derive these traits:

```rust
use serde::{Serialize, Deserialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
struct JobPayload {
    id: String,
    task_name: String,
    created_at: chrono::DateTime<chrono::Utc>,
}
```

## Message Lifecycle States

Messages transition through distinct states during their lifetime:

```
┌─────────────────────┐
│   Main Queue        │  Message is published and waiting
└──────────┬──────────┘
           │ consume()
           ↓
┌─────────────────────┐
│ Processing Queue    │  Message is being processed
└──────────┬──────────┘
           │
      ┌────┴────┐
      │          │
acknowledge()  reject()
      │          │
      ↓          ↓
 ┌────────┐  ┌──────────────────┐
 │  Done  │  │ Retry (if        │
 │(removed)  │ attempts remain) │
 └────────┘  │ or Failed Queue  │
             └──────────────────┘
```

### State Transitions

- **Consume**: Message moves from Main Queue → Processing Queue
- **Acknowledge**: Message removed from Processing Queue (successfully processed)
- **Reject**: Message moves to Failed Queue (if retry limit exceeded) or back to Main Queue (if retries remain)

## Attempt Tracking

The `attempts` field tracks how many times a message has been attempted:

```rust
let message = queue.consume::<JobPayload>("jobs", None).await?;

if message.attempts > 0 {
    println!("This is a retry. Attempt #{}", message.attempts);
}

// On first consumption, attempts is 0
// Each reject increments attempts by 1
// When attempts reaches the retry limit, message moves to dead-letter queue
```

### Configuring Retry Limits

```rust
use broccoli_queue::queue::{BroccoliQueue, RetryStrategy};

let queue = BroccoliQueue::builder("redis://localhost:6379")
    .failed_message_retry_strategy(
        RetryStrategy::new().with_attempts(5)
    )
    .build()
    .await?;
```

## Fairness with Disambiguator

The `disambiguator` field enables fair message processing across different categories:

```rust
#[derive(Clone, Serialize, Deserialize)]
struct OrderPayload {
    order_id: String,
    customer_id: String,
}

let payload = OrderPayload {
    order_id: "order-123".to_string(),
    customer_id: "cust-456".to_string(),
};

// Publish with a disambiguator to group messages by customer
queue.publish(
    "orders",
    Some("cust-456".to_string()), // disambiguator
    &payload,
    None,
).await?;
```

When consuming with fairness enabled, Broccoli ensures that messages from different disambiguators are processed in a balanced way, preventing one customer from monopolizing the queue.

## Serialization

Broccoli serializes message payloads to JSON internally. This means:

- Your payload must be JSON-serializable
- The `payload` field is stored as a JSON string in the broker
- Deserialization happens automatically when consuming
- Complex nested structures work fine as long as they're JSON-compatible

## Example: Complete Message Flow

```rust
use broccoli_queue::queue::BroccoliQueue;
use serde::{Serialize, Deserialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
struct EmailJob {
    recipient: String,
    subject: String,
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let queue = BroccoliQueue::builder("redis://localhost:6379").build().await?;

    let job = EmailJob {
        recipient: "user@example.com".to_string(),
        subject: "Hello".to_string(),
    };

    // Publish
    let message = queue.publish("email_jobs", None, &job, None).await?;
    println!("Published: {}", message.task_id);

    // Consume and process
    let consumed = queue.consume::<EmailJob>("email_jobs", None).await?;
    println!("Consumed: {} (attempt #{})", consumed.task_id, consumed.attempts);

    // Acknowledge on success
    queue.acknowledge("email_jobs", consumed).await?;

    Ok(())
}
```
